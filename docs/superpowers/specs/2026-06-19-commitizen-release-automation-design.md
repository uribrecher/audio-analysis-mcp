# Commitizen release automation — design

Issue: [#59](https://github.com/uribrecher/audio-analysis-mcp/issues/59) — "Adopt Commitizen for release automation (version bump + CHANGELOG from Conventional Commits)"

## Goal

Replace the hand-edited version + missing changelog with a **commit-derived** release flow built on
[Commitizen](https://commitizen-tools.github.io/commitizen/). `cz` infers the semver bump from
Conventional Commits, updates `[project].version`, regenerates `CHANGELOG.md`, and the version tag
publishes to PyPI. Releases stay **deliberate** — nothing publishes on a plain feature merge.

## Decisions (resolved with the maintainer)

1. **Release model: release-PR automation** (mirrors `keyboards-mcp`'s Changesets flow). After
   `feat:`/`fix:` commits accumulate on `main`, CI auto-opens a `chore(release): vX.Y.Z` PR carrying
   the version bump + regenerated changelog. Merging that PR is the deliberate release trigger.
2. **Conventional-Commit enforcement: PR-title gate only.** The repo squash-merges, so the commit
   that lands on `main` (and that `cz` reads) is the **PR title**. A `pull_request` check validates
   the title; individual branch commits are unconstrained. A per-commit `cz check` would be useless
   under squash-merge.
3. **CHANGELOG seeding: backfill.** `cz changelog` seeds `CHANGELOG.md` from existing Conventional
   Commits so the `0.1.0` `feat:`/`fix:` history is recorded from day one. Non-conventional historical
   commits are skipped (partial-but-accurate).
4. **Publish wiring: `workflow_dispatch`, no PAT, no reusable workflow.** PyPI Trusted Publishing
   **forbids reusable workflows** (warehouse#11096), so `publish.yml` stays a standalone workflow.
   A tag pushed by the built-in `GITHUB_TOKEN` does **not** trigger `publish.yml` — but
   `workflow_dispatch` is exempt ("workflow_dispatch and repository_dispatch events always create
   workflow runs"). So `release.yml` tags the merge commit and then **dispatches** `publish.yml`.
5. **PyPI `pypi` environment: keep, with a required reviewer.** Publishing pauses at the `pypi`
   environment for a manual approval click — a second deliberate gate on top of merging the release PR.

### Considered & rejected (from the issue)

- **Towncrier / scriv** — changelog-only, no version logic.
- **python-semantic-release / release-please** — bias toward release-on-merge; we want the explicit
  release-PR + approval gates.
- **Reusable `publish.yml` called by `release.yml`** — initially chosen, then **rejected**: PyPI
  Trusted Publishing does not support reusable workflows.
- **PAT to push the tag** — avoided; `workflow_dispatch` via `GITHUB_TOKEN` achieves the same with no
  long-lived secret.

## Current state (facts that shaped the design)

- `v0.1.0` is **already tagged and published on PyPI**. The 5 commits since are all `docs:`/`chore:`/
  non-conventional → `cz bump` correctly infers **no release** today. This is the verification anchor.
- The repo **squash-merges** (every `main` commit ends `(#NN)`).
- PyPI Trusted Publisher is registered as **`publish.yml`** (confirmed by maintainer) — matches the
  real workflow filename, so **no PyPI-side change is needed**.
- `publish.yml` triggers `on: push: tags: ['v*']`, builds with `uv build --no-sources`, `twine check`,
  then `pypa/gh-action-pypi-publish` in `environment: pypi` (OIDC trusted publishing).

## Components

### 1. `pyproject.toml`

```toml
[tool.commitizen]
name = "cz_conventional_commits"
version_provider = "pep621"      # reads/writes [project].version
tag_format = "v$version"          # must match publish.yml's tags: ['v*']
update_changelog_on_bump = true
major_version_zero = true         # stay in 0.x; breaking changes bump minor until a deliberate 1.0
```

Add `commitizen` to `[dependency-groups].dev`.

### 2. `CHANGELOG.md`

Seeded once via `uv run cz changelog` (backfill). Thereafter regenerated by `cz bump`.

### 3. `.github/workflows/pr-title.yml` — PR-title gate

- Trigger: `pull_request: [opened, edited, synchronize, reopened]`.
- Uses `amannn/action-semantic-pull-request@v5` with `GITHUB_TOKEN`.
- Allowed types: `feat, fix, docs, chore, refactor, perf, test, build, ci, style, revert`.
- Permissions: `pull-requests: read`.
- Added to `main` branch protection as a **required status check**.

### 4. `.github/workflows/release.yml` — release-PR + tag + dispatch

Trigger: `push: branches: [main]`. Permissions: `contents: write`, `pull-requests: write`,
`actions: write`. `concurrency` guard keyed on the ref to serialize runs.

Core rule — **"a version in `pyproject.toml` with no matching git tag means a release commit"**:

1. Checkout full history **and tags**; install `uv` + sync the `dev` group (provides `cz`).
2. Read `V = [project].version`.
3. **If tag `vV` does not exist** → the just-merged commit is the release commit:
   - Create + push tag `vV` (via `GITHUB_TOKEN` — for the git record).
   - `gh workflow run publish.yml --ref vV` → fires `publish.yml` (dispatch is `GITHUB_TOKEN`-exempt).
   - Stop.
4. **Else** (`vV` already tagged) → look for a pending release:
   - `cz bump --dry-run --yes` to compute the next version `W` from commits since `vV`.
   - **No eligible commits** → no-op.
   - **Eligible** → on branch `release/vW`: run `cz bump --yes --files-only --changelog` (updates
     `[project].version` + regenerates `CHANGELOG.md`, **no commit, no tag** — exact flag set
     verified in the plan, since `--files-only` changelog behavior is version-dependent), commit
     those files, force-push the branch, and open/update a PR titled **`chore(release): vW`** with
     the changelog delta in the body.

This is self-correcting and loop-free: after tagging `vV`, the next `release.yml` run finds `vV`
tagged and `cz bump` from `vV` sees only the `chore(release)` commit → no new release PR. The
`chore(release):` title satisfies the PR-title gate and never itself triggers a bump.

### 5. `.github/workflows/publish.yml` — minimal change

- **Add** `workflow_dispatch` (so `release.yml` can dispatch it). **Keep** `push: tags: ['v*']`
  as the manual escape hatch. Keep `environment: pypi`, `--no-sources` build, twine check, OIDC.
- Stays **standalone** (never `workflow_call`) so Trusted Publishing keeps working unchanged.

### 6. Repo settings (via `gh api`, after spec approval)

`main` is governed by an existing **repository ruleset** (id `15346518`), not classic branch
protection. It already enforces: **squash-only merges**, signed commits, linear history, 1 review +
thread resolution, and the CI checks (`test-and-lint`, `audit`, `packaging-smoke`). **Repository
admins bypass the ruleset `always`** (`actor_id: 5`). Changes:

- `squash_merge_commit_title = PR_TITLE`, `squash_merge_commit_message = PR_BODY` (repo setting) —
  the validated PR title becomes the `main` commit subject `cz` reads; the PR body becomes the
  commit body (clean changelog bodies + `BREAKING CHANGE:` footer support). Squash-only is already
  ruleset-enforced, so disabling merge/rebase at the repo level is optional cleanup only.
- Add `pr-title` to the ruleset's `required_status_checks` (alongside the existing CI contexts).
- `pypi` environment: add `uribrecher` (id `695383`) as a **required reviewer**.

**Bot release-PR + required checks (resolved):** a PR opened by the Actions bot via `GITHUB_TOKEN`
does **not** trigger `pull_request` workflows, so `pr-title`/CI never report on the release PR and a
required-check rule would normally block it. The existing **admin bypass** resolves this cleanly:
the maintainer merges the bot's release PR using their ruleset bypass (the deliberate release
action). No PAT or GitHub App is introduced. The release PR title (`chore(release): vX.Y.Z`) is
valid-by-construction, so the gate adds no value there anyway.

### 7. Docs

- `README.md`: new **Releases** section — write conventional PR titles → merge → release PR
  auto-opens → merge it → approve the `pypi` environment → published. Plus the manual escape hatch
  (`uv run cz bump` locally → `git push --follow-tags`).
- `CLAUDE.md`: short pointer to the release flow + "every PR title must be a Conventional Commit".

## End-to-end flow

```
feature PR (conventional title) --merge--> main
        │                                    │
   pr-title.yml gate                    release.yml: feat/fix since tag?
                                              │ yes
                                   opens/updates "chore(release): vX.Y.Z" PR
                                              │ (maintainer merges, deliberately)
                                              ▼
                                   release.yml: version untagged → push vX.Y.Z tag
                                              │  + gh workflow run publish.yml --ref vX.Y.Z
                                              ▼
                                   publish.yml → pauses at `pypi` env (approval) → PyPI
```

## Testing / verification

- `uv run cz bump --dry-run` → confirms **no bump** today (only `docs:`/`chore:` since `v0.1.0`),
  proving "no forced publish".
- `uv run cz changelog --dry-run` → sane changelog output before committing the seed.
- `uv lock` updates cleanly; `uv sync --dev` installs `commitizen`.
- Workflow YAML validated (e.g. `actionlint` if available); release.yml logic exercised by a dummy
  `feat:` on a throwaway branch is **out of scope** for the PR — verified by reasoning + first live run.
- Manual escape hatch documented and unchanged from the issue's original intent.

## Risks & mitigations

- **Bespoke `release.yml` is the main risk.** Mitigation: the single "untagged version = release"
  rule keeps it small and self-correcting; first real release watched closely; manual `cz bump`
  path remains as fallback.
- **PyPI env binding mismatch.** If PyPI's trusted publisher pins an environment name ≠ `pypi`,
  publishing fails with an OIDC error. Mitigation: workflow uses `environment: pypi`; surfaced on the
  first publish; PyPI config left untouched (already correct for the filename).
- **`actions: write` permission** on `release.yml` is required to dispatch `publish.yml` — scoped to
  that workflow only.

## Out of scope

- Migrating `keyboards-mcp` or `sound-recreation-agent` (TS, already on Changesets).
- A `1.0.0` cut (`major_version_zero = true` keeps us in `0.x`).
- Backfilling perfectly accurate history for non-conventional past commits.
